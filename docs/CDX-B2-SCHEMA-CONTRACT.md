# CDX-B2 Schema Contract

**Status:** reviewed normative M2-B contract cut point; production
implementation remains pending. This file is normative only together with
`docs/GOVERNED-MUTATION-BRIDGE-CONTRACT.md`,
`docs/MEMORY-AUTHORITY-CONTRACT.md`, and
`docs/GOVERNED-MUTATION-DISPOSITION-REGISTRY.md`.

CDX-B2 is a disjoint governance overlay for one ratified-erasure
specialization. CDX-M1 remains runtime and read authority; exact CDX-B1 stays
non-authoritative and byte-identical to its protected baseline. Nothing here
authorizes a provider call, dataset download, publication, or terminal store
deletion.

## 1. Canonical kernel configuration

markers below. It is not regenerated by enumerating an object. Its byte length
is exactly `5704`; its lowercase SHA-256 is
`e1ded27e33516d73c60da1f4a4c9cb0767b1bb0b1482e78b429449ec7c0b07f4`.
The JSON preserves the complete reference registry even though this milestone
emits only the ratified-erasure specialization.

Part 14 erasure reference. At that exact commit,
`docs/spec-palari-brain/parts/14-operations.tex` is blob
`8e08234eb822ac1c62d31ea9465226941226bc95`; that blob was checked as
non-config review provenance. It is not a new `KERNEL_CONFIG_JSON` field:
the Unified tree commit already pins it, and neither the transitive reference

From the repository root, this independently checks exact framing,
minification, ASCII, byte length, parse/serialize identity, and digest:

```bash
node --input-type=module <<'NODE'
import { createHash } from 'node:crypto'
import { readFileSync } from 'node:fs'

const source = readFileSync('docs/CDX-B2-SCHEMA-CONTRACT.md', 'utf8')
const begin = '<!-- KERNEL_CONFIG_JSON_BEGIN -->\n```json\n'
const end = '\n```\n<!-- KERNEL_CONFIG_JSON_END -->'
const start = source.indexOf(begin)
if (start === -1) throw new Error('missing config start marker')
const finish = source.indexOf(end, start + begin.length)
if (finish === -1) throw new Error('missing config end marker')
const json = source.slice(start + begin.length, finish)
if (/[^\x00-\x7f]/u.test(json)) throw new Error('config is not ASCII')
if (Buffer.byteLength(json, 'utf8') !== 5704) throw new Error('wrong byte length')
if (JSON.stringify(JSON.parse(json)) !== json) throw new Error('not canonical minified JSON')
const actual = createHash('sha256').update(json, 'utf8').digest('hex')
const expected = 'e1ded27e33516d73c60da1f4a4c9cb0767b1bb0b1482e78b429449ec7c0b07f4'
if (actual !== expected) throw new Error(`hash mismatch: ${actual}`)
console.log(`${actual} ${Buffer.byteLength(json, 'utf8')}`)
NODE
```

The schema constrains hashes to lowercase 64-hex. The runtime additionally
requires the singleton meta hash and every decision hash to equal the constant
above. A shape-valid different hash is invalid.

## 3. Exact trigger SQL

There are exactly eleven B2 triggers. Every one targets a B2 table and its
body contains only `SELECT`; no trigger performs B2, CDX-M1, or B1 DML. The
decision trigger checks sequence, configuration, the stream's first-decision
authority-ledger binding, nondecreasing time, and the pre-projection state. The
meta advance trigger checks the exact completed tail and post-projection state.

```sql
CREATE TRIGGER cdx_b2_meta_no_delete
BEFORE DELETE ON cdx_b2_meta
BEGIN
  SELECT RAISE(ABORT, 'cdx_b2_meta_no_delete');
END;

CREATE TRIGGER cdx_b2_meta_advance_guard
BEFORE UPDATE ON cdx_b2_meta
BEGIN
  SELECT CASE WHEN
    NEW.singleton IS NOT OLD.singleton OR
    NEW.schema_version IS NOT OLD.schema_version OR
    NEW.kernel_profile IS NOT OLD.kernel_profile OR
    NEW.projection_profile IS NOT OLD.projection_profile OR
    NEW.authority_profile IS NOT OLD.authority_profile OR
    NEW.stream_id IS NOT OLD.stream_id OR
    NEW.workspace_id IS NOT OLD.workspace_id OR
    NEW.checkpoint_id IS NOT OLD.checkpoint_id OR

The classification order is exact and first-match:

```text
target absent -> missing_target
else Palari or nullable user differs -> scope_mismatch
else shared = 1 -> shared_scope_unsealed
else any incident link -> incident_edges_unemittable
else private exact-scope zero-link target -> applied
```

Memory type and `current|ended` do not affect eligibility. Zero or multiple
FTS rows are projection corruption and roll back; they are never a committed
refusal. Applied effects are exactly ordinal zero atom removal and ordinal one
FTS removal. There is no link effect.

The first committed decision, whether applied or refused, binds its exact
`authority_ledger_id` to that B2 stream. Every later decision must match the
sequence-one row byte-for-byte. The insert trigger enforces that rule directly;
the complete reducer independently requires one exact ledger id across
`1..head`. A complete reopen with `head > 0` therefore retains that binding,
and any authority presented for a later governed write must name the same
ledger. Root/provider validation, error selection, and public construction
semantics remain owned by `docs/MEMORY-AUTHORITY-CONTRACT.md`.

## 4. Exact structural manifests

Before comparison, every non-NULL application-object `sqlite_schema.sql`
value and its reviewed expected statement are normalized by this exact local
B2 algorithm:

1. replace every CRLF pair (`0d 0a`) with LF (`0a`), leaving a lone CR and all
   other interior bytes unchanged;
2. trim from both ends only ASCII `09`, `0a`, `0b`, `0c`, `0d`, and `20`;
3. if and only if the resulting final character is one semicolon, remove that
   one semicolon; and
4. trim from the new right edge only the same six ASCII whitespace bytes.

There is no case folding, comment removal, quote rewriting, internal
whitespace collapse, repeated-semicolon removal, or Unicode whitespace trim.
The normalized values must then be exactly equal. Autoindexes are checked
separately as `sql IS NULL`. This algorithm is defined and implemented in the
B2 schema leaf; it does not import or call protected B1 normalization code.

The exact BINARY-sorted trigger target inventory is:

```text
cdx_b2_checkpoint_insert_guard | cdx_b2_legacy_checkpoint
cdx_b2_checkpoint_no_delete    | cdx_b2_legacy_checkpoint
cdx_b2_checkpoint_no_update    | cdx_b2_legacy_checkpoint
cdx_b2_decision_next_sequence  | cdx_b2_decisions
cdx_b2_decisions_no_delete     | cdx_b2_decisions
cdx_b2_decisions_no_update     | cdx_b2_decisions
cdx_b2_effect_insert_guard     | cdx_b2_effects
cdx_b2_effects_no_delete       | cdx_b2_effects
cdx_b2_effects_no_update       | cdx_b2_effects
cdx_b2_meta_advance_guard      | cdx_b2_meta
cdx_b2_meta_no_delete          | cdx_b2_meta
```

Verification rejects every unknown ASCII-case-folded `cdx_b2_*` object or
autoindex, every case-spoofed known name, every extra main trigger targeting a
B2 or CDX table, and every TEMP trigger. The A2 runtime trigger oracle changes
only by adding the exact eleven triples above. Exact B1 trigger triples remain
the only permitted non-CDX trigger additions.

## 5. Bootstrap and migration

Bootstrap is a serialized pre-handle control-plane transaction. Its exact
order is below. `bootstrapCdxB2InTransaction` executes steps 4--13 after the
bridge-owned coordinator has issued the active lease in step 3; the
coordinator alone performs transaction control and proves steps 1--3 and 14.
This function is the sole bootstrap-only owner of certified M0/M1 structural
completion/verification and the `CDX-B2` marker insert. It never performs
semantic CDX memory/link/FTS-row DML. Its other journal operations perform no
CDX or B1 DML.

1. Prove the native connection is open and not in a transaction.
2. Set and read back `foreign_keys=ON`, `busy_timeout=0`,
   `recursive_triggers=ON`, `ignore_check_constraints=OFF`, and
   `trusted_schema=OFF`.
3. Execute one `BEGIN IMMEDIATE` and prove exact open/active state.
4. Before any ordinary CDX repair or any B2 creation, enumerate folded B2
   names and inspect the `CDX-B2` migration marker. Classify only:
   - no B2 object and no marker: B2-absent candidate, not yet eligible for B2
     creation;
   - exact complete B2 manifest and exactly one marker: verify-only;
   - every other combination: invalid, rollback, never repair.
5. For a B2-absent candidate only, run the ordinary certified A2 CDX-M0/M1
   completion/repair path and then verify its exact completed M0/M1 schema,
   migrations, triggers, and FTS parity. Only that success promotes the state
   to a B2 initialization candidate. No B2 object or marker exists yet. A
   complete B2 state does not enter A2 repair, and a partial, extra, case-
   spoofed, or mutated B2 state rolled back in step 4 is never repaired even
   when ordinary M0/M1 would otherwise be repairable.
6. For a promoted B2 initialization candidate only, snapshot all memories in
   `id COLLATE BINARY` order, then all links in
   `id COLLATE BINARY` order. Capture only:
   - memory: `id,palari_id,user_id,type,shared,current|ended`;
   - link: `id,from_memory_id,to_memory_id`.
7. Sample one captured-native canonical `checkpoint_at` after the snapshot.
8. Generate one `b2s_` stream UUIDv4, then one `b2c_` checkpoint UUIDv4. A
   collision aborts; neither ID is retried.
9. Create the exact four tables, partial index, and eleven triggers without
   `IF NOT EXISTS`.
10. Insert the singleton meta row with head zero, exact snapshot counts, the
    final config hash, and the CDX memory-column-order variant mapped exactly
    to `cdx_m1_order_0|1|2`.
11. Insert one-based checkpoint ordinals: the complete memory block first,
    then the complete link block.
12. Insert the migration marker in the same transaction:

```sql
INSERT INTO memory_migrations(id, applied_at)
VALUES ('CDX-B2', :checkpoint_at);
```

13. Run complete verification and replay. A verify-only reopen branches
    directly from step 4 to this verification without executing steps 5--12.
14. Commit and prove exact open/inactive state before publishing a handle.

A complete reopen samples no clock, generates no ID, executes no B2
DDL/DML, and performs no repair. Its migration set is exactly `CDX-M0`,
`CDX-M1`, and `CDX-B2`; `CDX-B2.applied_at` must equal
`cdx_b2_meta.checkpoint_at`. The marker intentionally makes the certified A2
binary fail its exact two-row migration oracle on a later open. It cannot fence
an already-open old process or the old filesystem deletion function.

Any bootstrap commit-outcome uncertainty publishes no handle and follows the
existing poison/close proof law. Initialization never commits a partial B2
object set, checkpoint, or marker through a supported path.

## 6. Governed co-commit

The bridge is the sole production A1 owner. Within one A1 transaction it:

1. verifies B2 and the current CDX projection completely;
2. reads the verified head and, when `head > 0`, its exact tail decision under
   the write snapshot;
3. compares the already-authorized snapshot's `observedAt` with the tail's
   `observed_at`; an earlier value fails as `governance_clock_invalid`;
4. rejects historical reuse of either `authority_event_id` or
   `capability_id`;
5. generates the next decision id and patch id exactly once; collision is an
   error and is never retried inside the call;
6. runs exact C1--C8 admission and the exact six-field resolver on the one
   canonical ratified-erasure patch;
7. computes the first-match classifier;
8. inserts one decision at `head + 1`;
9. for an applied decision, inserts effect ordinal zero then ordinal one;
10. invokes the sole A2 projection applier for one branded
   `cdx_memory_delete`;
11. proves target memory and FTS absence and byte-identical link membership;
12. advances the singleton head by exactly one;
13. reruns complete verification and replay; and
14. exposes a result only after A1 proves commit.

Step 3 is skipped only at head zero; equality is permitted. It runs after the
complete verifier, so an invalid existing history wins, and before historical
nonce lookup or id generation, so clock rollback wins over reuse/collision and
samples no decision/patch id. On failure no decision is appended, A1 rolls
back, and the bridge and current authority retire under the exact
`governance_clock_invalid` settlement in
`docs/MEMORY-AUTHORITY-CONTRACT.md` and
`docs/GOVERNED-MUTATION-BRIDGE-CONTRACT.md`. The decision trigger's
nondecreasing-time predicate remains independent defense in depth for direct
SQL; the supported bridge detects the condition proactively and does not
surface its trigger abort as a generic journal failure.

A classified refusal inserts no effect and performs no CDX DML. Missing,
wrong-scope, shared, and incident-edge refusals consume the valid grant after
commit. Admission/resolver divergence, FTS cardinality mismatch, projection
mismatch, effect failure, postcondition failure, or verifier failure rolls the
whole transaction back and is not represented as a committed refusal.

The schema requires
`evidence_at <= issued_at <= effective_at = observed_at < expires_at`.
`evidence_at` is the patch provenance timestamp. `observed_at` is the native
use time and the admission `now`; issuance time is never substituted for
action time. Expiry equality refuses before A1.

## 7. Complete verifier and reducer

Verification order is fixed:

1. connection policy and exact current ownership: the bootstrap owner during
   open, or an active A1 lease during a governed transaction;
2. exact layout;
3. singleton, configuration, and migration;
4. checkpoint;
5. journal/reducer;
6. live projection.

The layout phase proves exact normalized stored SQL, application objects,
autoindexes, table xinfo, index list/xinfo, three B2 FKs, STRICT flags, trigger
names/targets/bodies, folded-name closure, and both
`PRAGMA main.foreign_key_check` and `PRAGMA main.quick_check` success.

The singleton phase requires exactly one row and every scalar literal above,
the exact runtime workspace, the canonical configuration hash, safe integer
counts/head, the exact detected CDX schema variant, and one canonical
`CDX-B2` marker whose time equals `checkpoint_at`.

Checkpoint verification requires:

- row count exactly memory count plus link count;
- ordinals exactly contiguous `1..N`;
- memories exactly ordinals `1..memory_count`;
- links exactly the remaining block;
- strictly increasing BINARY IDs inside each block;
- each row satisfies exactly one memory/link union;
- every checkpoint link endpoint resolves to a checkpoint memory; and
- no checkpoint row changes after initialization.

The pure reducer begins with a map of checkpoint memory descriptors and an
unchanged map of checkpoint link endpoints. It reads every decision in exact
sequence `1..head` and requires no tail beyond the head. For each decision it
rechecks every UUID, profile, patch scalar, authority scalar, evidence scalar,
canonical timestamp, time inequality, config hash, nondecreasing
`observed_at`, outcome/reason matrix, and effect matrix. At head zero there is
no authority-ledger binding. At positive head it takes the sequence-one
`authority_ledger_id` as the stream binding and requires exact equality for
every decision through `head`; zero, two, or more distinct ledger ids is
invalid. It recomputes the classification in this exact order:

```text
missing -> scope -> shared -> incident edge -> applied
```

A refusal must equal that recomputed reason and have zero effects. An applied
decision must be the clean branch and have exactly atom then FTS effects for
the target. Apply removes only the target memory from the reducer map. Links
never change. The applied-target partial unique index, reducer, and projection
comparison jointly reject target-ID reuse and a second erasure.

After validating a decision row, the reducer mechanically derives this exact
synthetic local authority use view; it accepts no caller-supplied action or
rationale:

```js
{
  authorityLedgerId: decision.authority_ledger_id,
  authorityEventId: decision.authority_event_id,
  capabilityId: decision.capability_id,
  timestamp: decision.observed_at,
  palariId: decision.palari_id,
  userId: decision.user_id,
  action: ACTION_BY_OPERATION[decision.operation],
  scope: {
    workspaceId: meta.workspace_id,
    palariId: decision.palari_id,
    userId: decision.user_id,
    targetId: decision.target_id,
  },
  rationale: RATIONALE_BY_OPERATION[decision.operation],
  decision: decision.outcome,
  reason: decision.reason_code,
}
```

The two maps are closed one-entry null-prototype records:

```text
ACTION_BY_OPERATION.atom_erase = erase_atom
RATIONALE_BY_OPERATION.atom_erase = user_erasure_request
```

Any missing or additional map key, nonliteral derived value, unknown operation,
or different result is verifier failure. This is a pure view definition, not a
new persisted or configuration field, so the canonical 5704-byte configuration
and its hash do not change. The view's exact external join key is
`(authorityLedgerId, authorityEventId, capabilityId)`. It remains the bounded,
non-hash-linked local materialized use view described by the authority
contract; it does not synthesize a `prev-hash` or claim a portable Part 12
ledger.

Projection verification compares the reduced state to CDX-M1:

- exact remaining memory ID set;
- exact checkpoint Palari, nullable user, type, shared bit, and
  `current|ended` class for every remaining memory;
- exact checkpoint link ID/from/to set;
- exactly one FTS membership for every remaining memory;
- no FTS membership for any erased memory; and
- the existing A2 full row/content FTS parity and CDX runtime manifest.

It never calls B1 replay and never treats B1 as evidence or authority.

The exact-runtime acceptance proof must commit a sequence-one zero-effect
decision under ledger A and advance the head, then attempt sequence two under
otherwise valid ledger B. The second insert must abort with
`cdx_b2_decision_next_sequence`, leave the head and tail unchanged, and permit
no advance. Repeating sequence two under ledger A must pass the sequence/ledger
guard (the test may roll that insertion back). A separate close/reopen case
must use a fresh root whose internally monotone `observedAt` is earlier than
the verified prior tail and prove exact `governance_clock_invalid`, zero new
B2/CDX state, bridge poison, and current-authority retirement before nonce
lookup or id generation.

## 8. Payload boundary

The checkpoint and journal are **payload-column-free**, not globally
content-free. They omit content, keywords, content hash, source-message ID,
source kind, extractor, fictional status, link relation, confidence,
importance, access/decay telemetry, memory and link creation timestamps,
exact validity timestamps, SQL, arbitrary evidence text, and proposal JSON.
The checkpoint preserves arbitrary legacy IDs and scope text, so a whole-
database byte scan is neither required nor truthful.

Tests place distinct canaries only in excluded content-bearing fields and
prove that no B2 column value contains those canaries. B2 replay can detect
membership, retained descriptor, link, effect, and FTS-membership divergence.
It cannot detect valid-value changes confined to any omitted field—including
`memories.fictional`, memory/link `created_at`, or link `relation`—reconstruct
content, authenticate an external event, or establish legacy rows as
canonical truth.

## 9. Honest enforcement limits

SQLite has no general commit hook. A raw SQL writer can commit a partial
decision/effect tail without advancing meta; the next complete verification
detects it, but a trigger cannot prevent that commit. A sufficiently
privileged raw writer can also forge a fully coherent journal and projection:
the same-file tables have no signature or external anchor. B2 therefore does
not claim hostile-host safety, external-ledger authenticity, cryptographic
audit, or complete authority-history reconstruction.

The trusted host's captured `checkGrantActive` predicate is the use-time
authority check. B2 records the asserted ledger/event/capability identities
and proves that the supported bridge co-committed them with the projection; it
does not prove that the external ledger event exists or that the trusted
predicate was honest.

Only the clean leaves of `D-02` and `D-03` may apply. Every other disposition
registry leaf refuses with zero CDX mutation. Only a valid, externally active,
matching erasure grant may reach the four recorded target refusals; every
other refusal is pre-gate and has no B2 write.
